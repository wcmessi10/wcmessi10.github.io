## 1. DB auto_increment

단점 : 분산 데이터베이스 환경에서 PK가 중복될 수 있기 떄문에 식별자의 유일성이 보장되지 않는다

- 여러 샤드에서 동일한 PK를 가지는 상황

단점 : 클라이언트 측에 노출하면 보안 문제

장점 : 간단하기 때문에 다음 상황에서 유리할 수 있다

- 보안적인 문제를 크게 고려하지 않는 상황
- 단일 DB를 사용하거나 애플리케이션에서 PK의 중복을 직접 구분하는 상황

보안적인 문제가 염려된다면

- PK는 DB auto_increment로 사용하되 애플리케이션에서의 식별자는 별도 유니크 인덱스 사용

## 2. 유니크 랜덤 문자열 또는 숫자

- UUID 또는 난수를 생성하여 PK 지정

장점 : 키 생성 방식이 간단하다

단점 : 랜덤 데이터로 인해 성능 저하

- Clustered Index는 정렬된 상태를 유지한다
- 데이터 삽입 필요한 인덱스 페이지가 가득 찼다면, B+ tree 재구성 및 페이지 분할로 디스크 I/O 증가
- PK를 이용한 범위 조회가 필요하다면, 디스크에서 랜덤 I/O가 발생하기 떄문에 순차 I/O보다 성능 저하

## 3. 유니크 정렬 문자열

장점 : 분산 환경에 대한 PK 중복 문제 해결 가능

장점 : 보안 문제 해결

장점 : 랜덤 데이터에 의한 성능 문제 해결

ex) UUID v7, ULID 등의 알고리즘

데이터 크기에 따라 공간 및 성능 효율이 달라진다

- Clustered Index는 PK를 기준으로 만들어진다
- Secondary Index는 데이터에 접근할 수 있는 포인터를 가진다

단점 : PK가 크면 클수록 데이터는 더 많은 공간을 차지, 비교 연산에 의한 정렬/조회에 더 많은 비용 소모

## 4. 유니크 정렬 숫자

장점 : 분산 환경에 대한 PK 중복 문제 해결 가능

장점 : 보안 문제 해력

장점 : 랜덤 데이터에 의한 성능 문제 해결

ex) Snokflake, TSID 등의 알고리즘

장점 : 문자열 방식보다 적은 공간 사용
